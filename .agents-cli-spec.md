# Agent Spec — Ledgr-QBS (ADK 2.0 target)

> Status: DRAFT for review. This is a planning document only — it changes no code, deps, or behavior.
> Current baseline: working Slack app + deterministic pipeline, 442 tests passing, on `google-adk 1.30.0`.

## Overview

Ledgr-QBS is a Slack-native accounting-document assistant for SG/MY bookkeeping. Each Slack
channel = one client (profile in Firestore: FYE month, COA, tax registration, accounting software).

The product is a **coordinator agent** that owns a client's whole book. Today it does one thing
(document → ledger). The ADK 2.0 target makes it a small team:

- **Coordinator (LlmAgent)** — routes each Slack message: a document to process, or a question to answer.
- **Document workflow (dynamic ADK 2.0 `@node` Workflow, resumable)** — the existing `pipeline.py`
  spine, promoted to a workflow: classify → extract → categorize (COA) → tax code → reconcile →
  route (FY) → consolidate workbook. Low-confidence items pause for human approval in Slack.
- **Q&A agent (LlmAgent, chat mode)** — answers questions over the client's books
  ("biggest expense?", "P&L for FY24", "are we near the GST threshold?").

### Phasing
- **Phase 1 (this migration):** Coordinator + Document workflow (dynamic, resumable, with HITL) + Q&A agent, on ADK 2.0.
- **Phase 2 (next):** Bank/Reconcile subagent (same node pattern), then Tax/GST.

## Why ADK 2.0

- **HITL** — `RequestInput` node pauses the workflow for a Slack Yes/No approval (no AI needed → auditable).
- **Long-running / resume** — `ResumabilityConfig(is_resumable=True)`; dynamic workflows auto-checkpoint,
  so a batch that waits overnight for approval or survives a crash resumes where it stopped.
- **Parallel** — process a document batch concurrently (`asyncio.gather`); on resume only failed nodes re-run.

## Decision rule (clarity): Agent vs Workflow — both, by layer

- **LlmAgent** (name + instruction + tools) = a *worker node*: language, judgment, conversation.
- **`@node` function** = deterministic code node: parse, compute, format, write.
- **`RequestInput`** = human-input node.
- **`Workflow`** = the orchestrator that wires nodes (order, branch, loop, parallel, pause/resume).
- Use a **dynamic** workflow (not a static graph) because an accounting flow has loops + branching.

## Example Use Cases

1. User uploads 5 bills → coordinator routes to Document workflow → classify/extract/categorize per doc
   (parallel) → 1 low-confidence doc pauses with a Slack approval card → on approval, write ledger +
   return FY workbook to the channel.
2. User asks "what was my biggest expense category last month?" → coordinator routes to Q&A agent → answer.
3. Crash/restart mid-batch → workflow resumes; already-processed docs are skipped (idempotent writes).

## Tools Required

- **Gemini** — dev = AI Studio (env switch, avoids Vertex quota); prod = Vertex (PDPA). Extraction reads
  the bill summary → SR/ZR ledger lines + reconcile guard.
- **Firestore** — per-channel client profile (FYE drives FY routing). Also session store (resumable).
- **Slack (Bolt / socket)** — trigger + HITL surface + result delivery.
- **pdfplumber / vision** — hybrid bank-statement extraction (digital vs scanned).
- **openpyxl** — FY workbook consolidation.

## Constraints & Safety Rules

- **Idempotent side-effects.** On resume, ADK runs tools *at least once, possibly more*. Every
  ledger write / Slack post / GST entry MUST be guarded against duplicates.
- **PDPA** — prod inference on Vertex (data residency); asia-southeast1.
- **Reconcile guard** — extracted totals must reconcile before a doc is written; else flag for human.
- **No silent miscategorization** — low confidence → HITL approval, not a guess.
- **Auditability** — control flow stays deterministic; LLM only at classify/extract/categorize nodes.

## Success Criteria

- Document workflow eval ≥ 0.9 vs ground truth (matches existing target); no regression vs the 442-test baseline.
- HITL approval round-trips through Slack and resumes correctly.
- A killed batch resumes with zero duplicate ledger rows.

## Migration plan (1.30.0 → 2.0), spec-first then staged

> Blast radius is contained: the live Slack path uses `pipeline.py`, NOT the `_run_async_impl`
> Acting/Constitution path. That breaker lives only in `agent.py` + `general_invoice_agent.py`.

0. **Spec** (this doc) — approve target + scope. *No code change.*
1. **Safety net** — `git init` + commit the green baseline (project is not yet under version control);
   do all 2.0 work on a branch. Live code stays on 1.30.0 until 2.0 is proven green.
2. **Decouple import** — stop `invoice_processing/__init__.py` eagerly importing `agent.py`, so importing
   `pipeline` doesn't load the legacy BaseAgents (which may fail to import under 2.0).
3. **Bump runtime** — `google-adk>=2.0`, `uv sync`; `uv run pytest` (442) must stay green.
4. **Re-architect** — promote `pipeline.py` to a dynamic `@node` Workflow: LlmAgent nodes (classify/
   extract/categorize) + function nodes (reconcile/route/consolidate) + `RequestInput` approval node;
   add `ResumabilityConfig`; make writes idempotent. Retire the custom-BaseAgent Acting stage into nodes.
5. **Eval guard** — `agents-cli eval run` + `eval compare` vs baseline; hold ≥ 0.9.
6. **Deploy** — Cloud Run, asia-southeast1 (HITL + resume need Cloud Run/GKE, not Agent Runtime).
